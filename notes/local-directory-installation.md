# Local Directory Installation Feature

## Overview

DHTI now supports installing elixirs and conches from local directories using the new `-l` (or `--local`) flag. This feature enables seamless integration with locally generated projects, eliminating the need to push to a Git repository before testing.

## Motivation

Previously, DHTI required elixirs and conches to be hosted in Git repositories for installation. This workflow was limiting for:
- Rapid prototyping and testing
- Local development iterations
- Projects generated by AI agent skills (elixir-generator, conch-generator)
- Private or sensitive projects not ready for Git

The local directory installation feature addresses these limitations by allowing direct installation from the filesystem.

## Usage

### Elixir Installation from Local Directory

```bash
# Install an elixir from a local directory
npx dhti-cli elixir install -l /path/to/elixir -n elixir-name

# Using relative path
npx dhti-cli elixir install -l ./my-elixir -n dhti-elixir-myproject

# With dry-run to preview changes
npx dhti-cli elixir install -l /path/to/elixir -n elixir-name --dry-run
```

**Parameters:**
- `-l, --local <path>`: Path to the local directory containing the elixir project
- `-n, --name <name>`: Name of the elixir (required)
- `--dry-run`: Preview changes without executing them

**Requirements:**
The local directory must contain a valid elixir project structure:
- `pyproject.toml` with project metadata
- `src/<elixir_name>/` directory with Python code
- `src/<elixir_name>/chain.py` with DhtiChain implementation
- `src/<elixir_name>/bootstrap.py` with configuration

### Conch Installation from Local Directory

```bash
# Install a conch from a local directory
npx dhti-cli conch install -l /path/to/conch -n openmrs-esm-dhti-myconch

# Using relative path
npx dhti-cli conch install -l ./my-conch -n openmrs-esm-dhti-myproject

# With dry-run to preview changes
npx dhti-cli conch install -l /path/to/conch -n conch-name --dry-run
```

**Parameters:**
- `-l, --local <path>`: Path to the local directory containing the conch project
- `-n, --name <name>`: Name of the conch (required)
- `--dry-run`: Preview changes without executing them

**Requirements:**
The local directory must contain a valid OpenMRS ESM module structure:
- `package.json` with project metadata
- `src/` directory with React components
- `src/index.ts` with module exports
- `src/routes.json` with route/extension definitions

## How It Works

### Elixir Installation

When you install an elixir from a local directory:

1. **Path Resolution**: The CLI resolves the path to an absolute path
2. **Copy Resources**: Basic elixir resources are copied to `~/dhti/elixir/`
3. **Update pyproject.toml**: The local path is added as a dependency source:
   ```toml
   [dependencies]
   "elixir-name"
   
   [tool.uv.sources]
   elixir-name = { path = "/absolute/path/to/elixir" }
   ```
4. **Update server.py**: Import statements and routes are added for the elixir
5. **Docker Build**: The elixir can be built into a Docker image with `npx dhti-cli docker -t elixir`

### Conch Installation

When you install a conch from a local directory:

1. **Path Resolution**: The CLI resolves the path to an absolute path
2. **Copy Resources**: Basic conch resources are copied to `~/dhti/conch/`
3. **Copy Conch**: The entire conch directory is copied to `~/dhti/conch/<conch-name>/`
4. **Update Configuration**:
   - `importmap.json`: Adds the conch import mapping
   - `spa-assemble-config.json`: Adds the conch as a frontend module
   - `routes.registry.json`: Registers the conch routes/extensions
   - `Dockerfile`: Updates with conch name and version
5. **Docker Build**: The conch can be built into a Docker image with `npx dhti-cli docker -t conch`

## Path Resolution

The CLI handles both absolute and relative paths:

- **Absolute paths**: Used as-is (e.g., `/home/user/projects/my-elixir`)
- **Relative paths**: Resolved relative to the current working directory (e.g., `./my-elixir` тЖТ `/home/user/cwd/my-elixir`)

This is important because:
- Python's `uv` (used for elixir dependencies) requires absolute paths in `pyproject.toml`
- Docker needs to access files during build
- The paths must remain valid across different execution contexts

## Integration with Agent Skills

The local directory installation feature is designed to work seamlessly with AI agent skills:

### Workflow Example

1. **Generate Elixir**: Use the elixir-generator skill to create a new elixir
   - Elixir is created in `./generated-elixir/`
   - Project structure is complete and ready to install

2. **Generate Conch**: Use the conch-generator skill to create a new conch
   - Conch is created in `./generated-conch/`
   - Project structure is complete and ready to install

3. **Install Both**: Use the local directory installation feature
   ```bash
   npx dhti-cli elixir install -l ./generated-elixir -n dhti-elixir-myproject
   npx dhti-cli conch install -l ./generated-conch -n openmrs-esm-dhti-myproject
   ```

4. **Start DHTI**: Build and start the complete stack
   ```bash
   npx dhti-cli docker -n myregistry/elixir:1.0 -t elixir
   npx dhti-cli docker -n myregistry/conch:1.0 -t conch
   npx dhti-cli docker -u
   ```

This workflow is orchestrated by the new **start-dhti** skill.

## Development Workflow

The local directory installation feature enables rapid development iteration:

### Initial Setup

```bash
# Generate or create your elixir
cd /path/to/projects
mkdir my-elixir && cd my-elixir
# ... develop elixir ...

# Install to DHTI
npx dhti-cli elixir install -l . -n dhti-elixir-myproject

# Build and start
npx dhti-cli docker -n test/elixir:1.0 -t elixir
npx dhti-cli docker -u
```

### Development Mode

For rapid iteration without rebuilding Docker images:

```bash
# Make changes to your elixir
cd /path/to/projects/my-elixir
# ... edit chain.py, bootstrap.py, etc. ...

# Copy updated files to running container
npx dhti-cli elixir dev -d . -n dhti-elixir-myproject
```

This copies the updated files directly to the running Docker container and restarts it.

### Rebuild for Permanent Changes

When you're ready to create a new image with your changes:

```bash
# Re-install with updated code
npx dhti-cli elixir install -l /path/to/my-elixir -n dhti-elixir-myproject

# Rebuild Docker image
npx dhti-cli docker -n test/elixir:2.0 -t elixir

# Update compose and restart
npx dhti-cli docker -d  # Stop old containers
npx dhti-cli docker -u  # Start new containers
```

## Comparison with Other Installation Methods

DHTI supports three installation methods for elixirs and conches:

### 1. Git Repository Installation

```bash
npx dhti-cli elixir install -g https://github.com/user/repo.git -n elixir-name
npx dhti-cli conch install -g https://github.com/user/repo.git -n conch-name
```

**Pros:**
- Works with remote repositories
- Supports branches and tags
- Supports subdirectories (monorepos)
- Suitable for published/shared projects

**Cons:**
- Requires pushing to Git before testing
- Not suitable for rapid iteration
- May require public repository or credentials

### 2. PyPI/npm Installation (Elixirs Only)

```bash
npx dhti-cli elixir install -p 'package-name = ">=1.0.0"' -n elixir-name
```

**Pros:**
- Works with published packages
- Standard Python package management
- Version constraints supported

**Cons:**
- Requires publishing to PyPI
- Not suitable for development/testing
- Not available for conches

### 3. Local Directory Installation (New)

```bash
npx dhti-cli elixir install -l /path/to/elixir -n elixir-name
npx dhti-cli conch install -l /path/to/conch -n conch-name
```

**Pros:**
- No Git repository required
- Rapid iteration and testing
- Works with AI-generated projects
- Supports private/sensitive projects
- Ideal for prototyping

**Cons:**
- Path must remain valid
- Not suitable for distributed deployment
- Requires local access to source code

## Best Practices

### Project Structure

Ensure your local directories follow the expected structure:

**Elixir:**
```
my-elixir/
тФЬтФАтФА pyproject.toml
тФЬтФАтФА README.md
тФЬтФАтФА src/
тФВ   тФФтФАтФА my_elixir/
тФВ       тФЬтФАтФА __init__.py
тФВ       тФЬтФАтФА chain.py
тФВ       тФЬтФАтФА bootstrap.py
тФВ       тФФтФАтФА ... (other modules)
тФФтФАтФА tests/
    тФФтФАтФА ... (test files)
```

**Conch:**
```
my-conch/
тФЬтФАтФА package.json
тФЬтФАтФА README.md
тФЬтФАтФА src/
тФВ   тФЬтФАтФА index.ts
тФВ   тФЬтФАтФА routes.json
тФВ   тФЬтФАтФА config-schema.ts
тФВ   тФФтФАтФА ... (components)
тФФтФАтФА ... (other files)
```

### Naming Conventions

Follow DHTI naming conventions:

- **Elixirs**: Name should start with `dhti-elixir-`
  - Example: `dhti-elixir-glycemic`, `dhti-elixir-medication-check`
  
- **Conches**: Name should start with `openmrs-esm-dhti-`
  - Example: `openmrs-esm-dhti-glycemic`, `openmrs-esm-dhti-chatbot`

### Path Management

- Use absolute paths for production installations
- Use relative paths for quick local testing
- Keep source directories stable (avoid moving/renaming after installation)
- Consider symlinking for development: `ln -s /path/to/dev ~/projects/dev`

### Version Control

Even when using local installation, maintain version control:

```bash
cd my-elixir
git init
git add .
git commit -m "Initial commit"
```

This allows you to:
- Track changes during development
- Revert to working states
- Eventually publish to a repository
- Collaborate with others

### Testing Before Installation

Always test before installing:

```bash
# Test elixir
cd my-elixir
uv run pytest

# Test conch
cd my-conch
yarn test
yarn build

# Dry-run installation
npx dhti-cli elixir install -l . -n dhti-elixir-myproject --dry-run
npx dhti-cli conch install -l . -n openmrs-esm-dhti-myproject --dry-run
```

## Troubleshooting

### Common Issues

**1. Path not found**

```
Error: ENOENT: no such file or directory
```

**Solution:**
- Verify the path exists: `ls -la /path/to/elixir`
- Use absolute paths: `npx dhti-cli elixir install -l $(pwd) -n name`
- Check for typos in path

**2. Invalid project structure**

```
Error: pyproject.toml not found
Error: routes.json not found
```

**Solution:**
- Verify the project structure matches expected format
- Check that all required files exist
- Use a template or cookiecutter for scaffolding

**3. Import errors after installation**

```
ImportError: cannot import name 'DhtiChain'
```

**Solution:**
- Verify the path in pyproject.toml is correct
- Check that the elixir directory structure is correct
- Ensure `__init__.py` files exist
- Try reinstalling: `npx dhti-cli elixir install -l /path -n name`

**4. Conch not visible in OpenMRS**

**Solution:**
- Check `routes.json` is properly configured
- Verify extension registration in `index.ts`
- Check browser console for errors
- Verify Docker container rebuilt: `docker compose logs dhti-frontend-1`

### Debugging

Enable detailed logging:

```bash
# Check what files would be modified
npx dhti-cli elixir install -l /path -n name --dry-run

# Check Docker container logs
docker compose logs dhti-langserve-1  # Elixir
docker compose logs dhti-frontend-1   # Conch

# Check installed files
ls -la ~/dhti/elixir/
ls -la ~/dhti/conch/

# Verify configuration
cat ~/dhti/elixir/pyproject.toml
cat ~/dhti/conch/def/importmap.json
```
